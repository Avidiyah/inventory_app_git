## What shipped

Schema foundation only — three new tables, one migration. **No services,
routes, UI, or behavior change.** Loading still writes ordinary `dispense`
rows in `transactions`; that wiring arrives in Phases 4–5.

### Files

- `backend/app/models.py` — added `MassStage`, `MassStageRoom`, `MassStageItem`
  ORM classes; widened the `sqlalchemy` import to include `Integer`,
  `UniqueConstraint`, `Index`, `text`.
- `backend/alembic/versions/b1f3d5a7c9e2_add_mass_staging_tables.py` — new
  migration, `down_revision = "a7c9e1f3b5d2"` (was head).

### Tables

**`mass_stages`** — `id` (UUID PK), `building_name` (Text), `status` (Text,
default `planning`, server_default `planning`), `created_by_id` (UUID, plain
nullable FK→`users.id`), `created_at`, `updated_at` (ORM `onupdate`),
`completed_at?`.
- Partial unique index **`uq_mass_stages_active_building`** on `building_name`
  `WHERE status <> 'completed'` → one active stage per building, enforced in DB.

**`mass_stage_rooms`** — `id`, `stage_id` (FK→`mass_stages` `ON DELETE CASCADE`),
`room_number` (Text), `work_order_number` (Text), `sort_order` (Integer),
`created_at`.
- `UNIQUE(stage_id, room_number)` = `uq_mass_stage_rooms_stage_room`.
- Index `ix_mass_stage_rooms_stage_id`.

**`mass_stage_items`** — `id`, `room_id` (FK→`mass_stage_rooms` `ON DELETE
CASCADE`), `item_id` (plain FK→`items`), `planned_quantity` (Numeric),
`loaded_quantity` (Numeric, default/server_default `0`), `returned_quantity`
(Numeric, default/server_default `0`), `created_at`.
- `UNIQUE(room_id, item_id)` = `uq_mass_stage_items_room_item`.
- Indexes `ix_mass_stage_items_room_id`, `ix_mass_stage_items_item_id`.

### Design choices realized

- `status` is **Text** (not a DB enum), matching `User.role` /
  `Transaction.transaction_type`; the vocabulary is validated in the Phase 3
  domain module.
- `item_id` FK is **plain** (not cascade): a referenced item cannot be hard
  deleted — benign, since items are soft-deleted via `items.archived_at`.
- Cascade is used only for **owned plan data** (stage→rooms→items), never for
  audit data (`transactions` is untouched).
- The `transactions` table was **not** modified — load writes normal `dispense`
  rows later; the stage tables only hold plan + actuals.

### Deviations from plan

None. Revision id `b1f3d5a7c9e2` was hand-assigned (consistent with the repo's
hand-written migrations) rather than autogenerated.

## Verification (all green)

Run from `backend/` with the venv:

1. `python -c "import app.models"` → imports OK; `mass_stages`,
   `mass_stage_rooms`, `mass_stage_items` present in `Base.metadata`.
2. `alembic upgrade head` → `a7c9e1f3b5d2 → b1f3d5a7c9e2`, exit 0.
3. Index inspection → all three tables created with the expected unique +
   non-unique indexes, including the partial unique `uq_mass_stages_active_building`.
4. `alembic downgrade -1` (tables dropped) then `alembic upgrade head`
   (recreated) → clean round-trip.
5. `pytest` → 60 passed (existing suite unaffected).
6. Partial-index invariant (rolled-back tx): a 2nd `planning` stage for the same
   building is rejected; a `planning` stage **is** allowed alongside a
   `completed` one for the same building.
